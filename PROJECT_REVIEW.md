# Farm / AgroPath KG — подробное описание проекта для тезиса и презентации

## 1) О чём проект (идея и цель)

**Farm (AgroPath KG)** — это backend-платформа маркетплейса фермерской продукции с модулем
**оптимизации маршрута закупки/доставки**.

Главная идея:
- покупатель хочет купить товары у нескольких фермеров;
- система помогает не только оформить заказ, но и **построить более выгодный маршрут**;
- фермеры управляют товарами и обработкой заказов;
- администратор контролирует качество и доступы.

### Проблема, которую решает проект
В обычном сценарии пользователь (или курьер) ездит по фермерам в произвольном порядке,
что увеличивает:
- расстояние,
- время,
- расходы на топливо,
- риск задержек на плохих дорогах.

Проект переводит эту задачу в **формальную оптимизационную модель** и даёт сравнение
нескольких профилей маршрута (дешевле, быстрее, сбалансированно).

---

## 2) Практическое применение

Проект может использоваться как:
- **B2C-маркетплейс** локальных фермерских продуктов;
- **B2B-сервис** для закупщиков/магазинов, которые собирают партии у нескольких поставщиков;
- **логистический помощник** для небольших хозяйств и кооперативов;
- база для мобильного приложения (Flutter/Web), где backend уже готов для API-интеграции.

Экономическая ценность:
- уменьшение пробега;
- снижение расхода топлива;
- повышение предсказуемости доставки;
- прозрачная аналитика по маршрутам.

---

## 3) Технологический стек

По коду проекта:
- **Python + Django** (серверная логика и ORM);
- **Django REST Framework** (REST API);
- **JWT** (авторизация токенами);
- **PostgreSQL** (основная БД);
- **drf-yasg / Swagger / ReDoc** (документация API);
- **django-filter** (фильтрация/поиск/сортировка);
- **Google OR-Tools** (оптимизация TSP при наличии);
- **OSRM** (реальные дорожные расстояния), fallback на Haversine.

---

## 4) Архитектура проекта

Проект разделён на 4 доменных приложения:

1. **accounts** — пользователи, роли, профиль фермера, верификация email.
2. **market** — категории и товары (каталог, фильтры, CRUD).
3. **orders** — заказы, статусы, отзывы, лайки.
4. **routing** — построение/сравнение маршрутов и многокритериальная оценка.

Это модульная структура: каждое приложение отвечает за собственную бизнес-область,
что упрощает сопровождение и расширение.

---

## 5) Роли пользователей и права

В системе выделены роли:
- **customer** (покупатель),
- **farmer** (фермер),
- **admin** (администратор).

### Что может customer
- регистрироваться и входить;
- просматривать каталог;
- создавать заказы;
- оставлять отзывы и лайки;
- запрашивать оптимизацию маршрута по выбранным товарам.

### Что может farmer
- создавать/редактировать свои товары;
- управлять статусами заказов, где есть его товары;
- указывать координаты фермы в профиле (важно для маршрутизации).

### Что может admin
- расширенные права доступа;
- модерация и администрирование через Django Admin;
- контроль верификации/данных.

## 6) Основные функциональные блоки

## 6.1 Авторизация и аккаунты

Реализовано:
- регистрация пользователя;
- email-верификация кодом активации;
- JWT login (access + refresh);
- endpoint «текущий пользователь»;
- смена роли customer/farmer;
- обновление профиля фермера (название, адрес, координаты, параметры доставки).

Плюс дипломной части: это демонстрирует полноценный user-flow, а не «только алгоритм».

## 6.2 Каталог товаров (market)

Реализовано:
- категории (slug-based lookup);
- товары с привязкой к владельцу (фермеру);
- цена, вес, описание, изображение;
- фильтрация/поиск/сортировка;
- пагинация;
- endpoint «мои товары» для фермера.

Доступы:
- просмотр каталога открыт;
- создание — только фермер;
- изменение/удаление — владелец или админская роль.

## 6.3 Заказы, отзывы, лайки (orders)

Реализовано:
- создание заказа;
- просмотр списка и деталей заказа с учётом роли;
- изменение статуса заказа фермером (confirmed/delivering/done/cancelled);
- отзывы к товарам;
- запрет оставить отзыв на собственный товар;
- лайки и список избранных товаров.

Это важный бизнес-контур «после каталога»: взаимодействие пользователей и жизненный цикл сделки.

## 6.4 Маршрутизация и оптимизация (routing)

Самый «научный» модуль для тезиса.

Реализовано:
- сбор координат фермеров по выбранным product_ids;
- проверка валидности данных (товары активны, координаты есть);
- построение базового и оптимизированного порядка обхода;
- сравнение профилей оптимизации;
- возврат геометрии маршрута для визуализации на карте.

Профили:
- cheapest,
- fastest,
- balanced.

## 7) Математическая модель (ключ для комиссии)

В проекте задана многокритериальная функция:

**F(route) = w1·D + w2·C + w3·T + w4·R**

Где:
- **D** — расстояние,
- **C** — стоимость топлива,
- **T** — время,
- **R** — штраф за качество дороги.

Особенности реализации:
- дорожные расстояния и длительность берутся из **OSRM**;
- при недоступности OSRM используется fallback (Haversine);
- для поиска порядка точек используется **OR-Tools TSP solver**;
- если OR-Tools недоступен, есть fallback на nearest neighbor.

Таким образом, система устойчива к ограничениям среды и сохраняет работоспособность.

## 9) Что говорить на защите (готовый narrative)

Короткий скрипт:

1. **Проблема:** закупка у нескольких фермеров часто не оптимальна по маршруту.
2. **Цель:** создать сервис, который объединяет маркетплейс и оптимизацию логистики.
3. **Решение:** backend на Django/DRF + модуль маршрутизации с многокритериальной оценкой.
4. **Результат:** пользователь получает сравнение профилей и может выбрать лучший вариант
   (дешевле/быстрее/сбалансировано).
5. **Практическая польза:** снижение расходов и времени, повышение качества планирования.

## 11) Ограничения и планы развития

Что честно обозначить комиссии:
- требуется усиление автоматизированных тестов;
- прод-настройки безопасности нужно ужесточать (CORS и т.п.);
- желательно добавить наблюдаемость (structured logging, метрики);
- полезно провести серию экспериментов на реальных датасетах/маршрутах.

Планы развития:
- расчёт с учётом окон времени, пробок и вместимости;
- батч-оптимизация для нескольких заказов;
- дашборд KPI (экономия км/топлива/времени);
- мобильное клиентское приложение поверх текущего API.
